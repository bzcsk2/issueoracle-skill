## 一、项目定位与核心价值

IssueOracle 是一个 **Skill-first** 本地优先的代码审查工具链。它从 GitHub 上已修复的 issue 中挖掘 bug 模式，将这些「开源缺陷经验」结构化后用于审查本地代码，每个 finding 都带有具体的 OSS 证据链接。

**核心护城河**：不是 linter，不是通用 reviewer，而是「开源缺陷经验记忆层」——让真实世界的 bug 修复变成可复用的审查模式。

**六要素原则**：IssueOracle 只在以下条件全部满足时才报告 finding：

1. 匹配到结构化 bug pattern
2. 有本地 file/line 证据
3. 有合理的触发条件
4. 有置信度评分
5. 有误报边界说明
6. 来自公开 GitHub 数据时有 OSS 证据链接

> Never claim code is buggy only because similar projects had similar issues.

## 二、版本演进总览

```
v0.1 — 基础能力
  ├── Review 管线（本地 pattern 匹配）
```

## 三、设计决策摘要

| 决策点 | 选择 | 说明 |
|---|---|---|
| Python 最低版本 | **3.12+** | 需要 tomllib、type parameter syntax |
| Schema 校验 | **Pydantic v2** | 自动类型校验、嵌套模型、序列化 |
| Pattern Pack 语言 | **Python + TypeScript 先行** | FastAPI/SQLAlchemy + Express |
| 存储方案 | **文件系统**（~/.issueoracle/） | JSON/Markdown，零额外依赖 |
| Skill 形态 | **Skill-first** | SKILL.md 合约 + Python engine |
| GitHub 认证 | **GITHUB_TOKEN（可选）** | 无 token 60 req/hr；有 token 5000 req/hr |
| 运行时依赖 | **Pydantic + PyYAML** | GitHub API 用 stdlib urllib |
| 匹配引擎 | **三段式**（metadata → signal → trigger） | v0.1 确立，v0.2 不动 |
| 评分公式 | **五维加权**（0.25+0.25+0.20+0.15+0.15） | v0.1 确立，v0.2 不动 |
| 经验文档格式 | **双落盘**（.md + .json） | md 给人/agent 读，json 给 review 引擎用 |

### 5.2 三命令流水线

```text
scan ./my-project
  → 项目画像（语言/框架/功能类型/风险面）
  → 推荐 5 个同类型 OSS 项目（GitHub topic + stars）
  → 输出 scan-report.md + 推荐项目清单

mine owner1/repo1,owner2/repo2,...
  → 对每个仓库：搜 closed bug issues → 过滤 → 找 linked PR → 抽取 bug 经验
  → 汇总成叙事型 bug 经验 markdown
  → 落盘 ~/.issueoracle/bugplay/bug-experience.md + experience.json

review ./my-project --experience ~/.issueoracle/bugplay/bug-experience.md
  → 加载内置 packs + 经验文档 patterns
  → 索引本地代码 → 三段式匹配 → Finding 生成
  → 输出带 file/line/confidence/evidence 的报告
```

### 5.3 SKILL.md 合约要点

SKILL.md 是运行时的唯一真相源，当前版本 **v0.2.0**。

- **Intent 分类**：SCAN_PROJECT | REVIEW_REPO | REVIEW_DIFF | MINE_REPO | REVIEW_WITH_EXPERIENCE | VALIDATE_PACK | EXPLAIN_FINDING | HELP
- **输出契约**：每个 finding 必须包含 file/line、confidence、matched pattern、trigger condition、local evidence、OSS evidence、suggested fix、false-positive boundary
- **安全红线**：不输出低置信 finding，不输出无行证据的 finding，不输出原始 issue body/PR diff
- **Runtime preflight**：Python 3.12+、SKILL_DIR 解析、ISSUEORACLE_HOME 初始化、git 检查、不传本地代码到远端 LLM

## 六、命令体系详解

### 6.1 scan — 项目画像 + 相似项目发现

```bash
issueoracle.py scan <repo_path> [--emit markdown|json] [--max-repos 5] [--save-dir] [--debug]
```

**内部流程**：

```text
1. profile_repo(repo_path)          → RepoProfile（语言/框架/依赖/风险面）
2. classify_project_type(profile)   → "web_api" | "cli" | "frontend" | "library"
3. infer_search_topics(profile)     → ["fastapi", "web-framework", "rest-api"] ...
4. search_similar_repos(lang, topics, token, max)
   → GitHub /search/repositories?q=language:Python topic:fastapi stars:>100 sort:stars
   → 过滤 archived / <100 stars
   → 返回 RepoCandidate 列表
5. render_scan(profile, candidates) → scan-report.md
```

**项目类型分类规则**（`classify_project_type`）：

| 类型 | 判断关键词 |
|---|---|
| web_api | fastapi, flask, django, express, next, koa, starlette |
| cli | click, typer, commander, yargs, cobra, clap |
| frontend | react, vue, angular, svelte（且无 express/fastapi） |
| web_api（兜底） | risk_surfaces 含 "web" |
| library | 以上都不匹配 |

**输出格式**：

````markdown
# IssueOracle Scan Report

## Project Profile
- **Languages**: Python
- **Frameworks**: FastAPI, SQLAlchemy
- **Project type**: web_api
- **Risk surfaces**: web, database, auth
- **Dependencies**: fastapi, sqlalchemy, uvicorn

## Recommended Similar Projects
| # | Repo | Stars | Description |
|---|------|-------|-------------|
| 1 | [tiangolo/fastapi](url) | 75k | ... |

## Next Step
issueoracle.py mine tiangolo/fastapi,...
````

### 6.2 mine — 批量挖取 + 叙事型 bug 经验文档

```bash
issueoracle.py mine <owner/repo,...> [--max-issues 30] [--emit markdown|json] [--save-dir] [--debug]
```

**支持逗号分隔多个仓库**，内部循环处理每个仓库。

**内部流程（每个仓库）**：

```text
1. github_search.search_closed_issues(repo, label, max, token) → GitHubIssue[]
2. issue_filter.filter_issues(issues)                          → bug_issues[]
3. 对每个 bug_issue:
   a. evidence_linker.link_issue_to_prs(issue, repo, token)   → LinkedPR[]
   b. pattern_extract.extract_candidate(issue, prs, repo)    → CandidatePattern
      - _extract_root_cause：优先从 PR title（"fix: ..."），否则 issue title
      - _extract_signals：从 body 代码块 + issue 标题 + PR 标题关键词（v0.2 增强）
      - _extract_fixes：从 PR title
      - _infer_triggers：从 signals + issue title 推断
      - _classify_bug_type：按标题/信号关键词分类
4. 汇总所有仓库的所有 candidates
5. experience.aggregate(candidates, repos, total, bugs) → ExperienceReport
6. experience.to_markdown(report) → 叙事 markdown
7. store.save_experience(md, json) → ~/.issueoracle/bugplay/
```

**signals 抽取增强（v0.2）**：

v0.1 只从 issue body 代码块提取关键词。v0.2 合并三个来源：
1. issue body 中的代码块
2. issue 标题
3. PR 标题

通过 `_extract_signals_from_text()` 统一提取（90+ 个关键词覆盖 await/session/query/cors/auth/cache/serialize 等）。

**叙事型经验文档格式**（六要素）：

````markdown
# Bug Experience Report
Mined from: fastapi/fastapi, encode/starlette
Date: 2026-06-16
Total issues: 45 | Bug issues: 23

## Resource Leak (4 bugs)

### 1. Async session leak on exception path
- **Symptom**: connection pool exhaustion under concurrent requests
- **Root cause**: Async SQLAlchemy session not closed when exception raised
- **Trigger condition**: async request handler + DB session + exception path
- **Bad code signals**: `await`, `session`, `query(`
- **Fix**: use request-scoped dependency + `async with` context manager
- **Evidence**: [fastapi/fastapi#123](url)
````

按 bug_type 分组（resource_leak / sql_injection / misconfiguration / ...），agent 读完后知道「什么场景警惕什么 bug」。

**双落盘**：
- `~/.issueoracle/bugplay/bug-experience.md` — 叙事型，给人/agent
- `~/.issueoracle/bugplay/experience.json` — 结构化，给 review 引擎

### 6.3 review — 用经验回扫本地代码

```bash
issueoracle.py review <repo_path> [--experience <path>] [--changed] [--base main]
  [--severity-threshold medium|low|high] [--max-findings 20]
  [--emit markdown|json|compact] [--packs <dir>] [--save-dir] [--debug]
```

**Pattern 来源（双通道合并）**：

```text
1. 内置 packs/（Python FastAPI 5 个 + TypeScript Express 2 个）  ← pack_loader
2. --experience 指向的经验文档                                  ← experience.load_as_patterns()
3. 合并 → 统一 pattern 集合 → 走现有匹配引擎
```

**经验文档 → Pattern 转换逻辑**（`experience.load_as_patterns`）：

```text
- 优先读 .json（结构化可靠）
- 回退读 .md（正则解析 "### N. title" + "- **Key**: Value"）
- 每个 BugExperience → Pattern:
  - id: "exp-{原始id}"
  - language / frameworks / bug_type / confidence：直接取
  - bad_code_signals：直接取
  - trigger_conditions：从 trigger_condition 构造
  - evidence：保留 OSS 链接
  - false_positive_boundary：标注 "Auto-generated from mined experience"
```

**Review 管线**（v0.1 确立，v0.2 不改）：

```text
1. pack_loader.load_pack_dir(packs)           → Pattern[]
2. [+ experience.load_as_patterns(path)]       → Pattern[]（v0.2 新增）
3. profile.profile_repo(repo_path, changed)   → RepoProfile
4. code_index.index_repo(repo, profile)         → CodeChunk[]
   - Python: ast.parse → function/class/async function
   - TypeScript: 正则 → function/class/export
   - 其他: 滑动窗口 fallback（50 行窗口，10 行步长）
5. pattern_match.match(chunks, patterns, prof)  → MatchResult[]
   - Stage 1: metadata 召回（语言/框架/bug_type 匹配）
   - Stage 2: static signal 匹配（bad_code_signals 交叉）
   - Stage 3: trigger 判断（trigger_conditions 覆盖度）
   - 评分: 0.25×confidence + 0.25×code_match + 0.20×evidence
         + 0.15×trigger_coverage + 0.15×signal_strength
6. review.build_findings(matches, threshold)  → (Finding[], suppressed[])
   - 硬规则: 缺 file/pattern/trigger 任一字段 → suppress
   - 置信度阈值: low=0.3 / medium=0.5 / high=0.7
   - 按 confidence 降序，截断至 max-findings
7. render_report → markdown / json / compact
```

## 七、数据模型（schema.py 全景）

```
schema.py（15 个 Pydantic BaseModel）
│
├── Pattern 层
│   ├── OssEvidence          repo/issue/pr/commit/url/strength
│   ├── TriggerCondition     description + code_signals
│   ├── Pattern              ★ 核心模型（id/title/language/frameworks/bug_type/
│   │                            severity/symptoms/root_cause/triggers/signals/
│   │                            fixes/evidence/confidence/false_positive_boundary）
│   └── CodeChunk             file/start_line/end_line/symbol/language/imports/signals
│
├── Review 层
│   ├── RepoProfile           repo_path/languages/frameworks/dependencies/risk_surfaces/changed
│   ├── LocalEvidence         line + description
│   ├── Finding               id/severity/confidence/file/start-end/matched_pattern/
│   │                          trigger/local_evidence/oss_evidence/fix/validation/boundary
│   └── ReviewReport          version/scope/summary/findings/suppressed
│
├── Mining 层
│   ├── GitHubIssue           number/title/state/labels/body/url/dates/author
│   ├── LinkedPR              number/title/state/merged/url/commit_sha/files_changed
│   ├── CandidatePattern       ≈ Pattern + source_issue/source_repo/status="candidate"
│   └── MiningResult          repo/mined_at/issues_searched/kept/with_pr/candidates
│
├── Scan 层（v0.2 新增）
│   ├── ProjectProfile        repo_path/languages/frameworks/dependencies/
│   │                          risk_surfaces/project_type/search_topics
│   └── RepoCandidate         owner_repo/stars/description/url/topics
│
├── Experience 层（v0.2 新增）
│   ├── BugExperience         id/title/symptom/root_cause/trigger_condition/
│   │                          bad_code_signals/fix/evidence/bug_type/language/
│   │                          frameworks/confidence
│   └── ExperienceReport      source_repos/mined_at/total_issues/bug_issues/experiences
│
└── Validation 层
    └── ValidationResult      pack_path/patterns_valid/invalid/errors
```

**关键校验规则**：
- Pattern 的 `evidence` 不能为空（`@field_validator`）
- `confidence` 必须在 [0.0, 1.0] 范围
- `Severity` 仅允许 "low" | "medium" | "high" | "critical"

## 十、安全设计

### 10.1 GitHub 内容安全（safety.py）

GitHub issue/PR 内容被视为**不可信输入**：

| 防护措施 | 实现 |
|---|---|
| Prompt injection 防护 | 检测 `<system>` 标签、"ignore previous instructions" 等模式 → 替换为 `[redacted]` |
| Shell 代码块截断 | `bash/sh/shell/cmd/powershell` 代码块 → `[redacted shell block]` |
| 内容长度限制 | issue body 截断至 2000 字符 |
| 证据链安全 | 只保留链接和元数据（repo/issue#/pr#），不保留原文 |

### 10.2 本地代码安全

| 规则 | 实现 |
|---|---|
| 不上传本地代码 | 默认禁止远端 LLM，除非 `ISSUEORACLE_ALLOW_REMOTE_LLM=1` |
| 不执行外部命令 | mining 只读 GitHub API，不执行 issue 中的任何命令 |
| 最小权限 | 只读仓库文件 + 只调 GitHub REST API |

## 十四、已知偏差与后续建议

### 14.1 已知偏差（v0.2 验收中发现的低优先级问题）

| # | 计划要求 | 实际情况 | 影响 | 建议 |
